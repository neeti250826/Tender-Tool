KSA Awarded Tenders Scraper Spec

Scraper-ready extraction specification for Saudi Arabia public tender award data from NUPCO and Etimad, normalized into a shared output schema.

Source 1: NUPCO Source 2: Etimad Country: Saudi Arabia

Scope

This spec defines extraction rules for two public procurement sources:

  1. nupco
  2. tenders.etimad.sa

Normalization Rules

  • country: always Saudi Arabia
  • country_code: always SA
  • scraped_at_utc: ISO-8601 UTC timestamp at row creation time
  • publication_date: use YYYY-MM-DD when date-only
  • closing_date: store YYYY-MM-DD for Etimad; keep normalized date-only values for NUPCO as well
  • currency: set to explicit source currency when present; for these two sources, use SAR
  • description: prefer tender-level description; fallback to item_description
  • Filter both sources to the inclusive window 2024-01-01 through 2026-05-19 using the source-specific primary date field

Source 1: NUPCO

Discovery

Seed URL: https://www.nupco.com/en/tenders/

The listing page uses scrolling to reveal more tenders. The scraper must keep scrolling until no new tender cards load.

Award Data Location

Award data is attached to detail pages as PDFs rather than reliable structured HTML.

Typical labels include Final Result and Tender Preliminary Result.

NUPCO Detail Extraction

  • notice_id: tender ID from detail page
  • title: tender title or PDF TENDER NAME
  • publication_date: Bid Opening Date
  • closing_date: Submission Deadline
  • notice_url: detail page URL
  • status: result stage from listing/detail
  • classification: same as result stage unless a better field is exposed
  • buyer: NUPCO
  • awarding_agency_name: NUPCO
  • amount: Tender Booklet Price
  • Apply the date filter on normalized publication_date before PDF-heavy parsing continues
  • Emit progress/debug logs for listing discovery, retries, tender processing, PDF buckets, skips, and running row totals

NUPCO PDF Parsing Strategy

  1. Download all result PDFs from each tender detail page.
  2. Prefer Final Result over Preliminary Result if both exist.
  3. Extract text page by page and reconstruct rows from table layout.
  4. Do not use naive line splitting; rows can wrap across lines.
  5. Emit one row per awarded line item per supplier.
  6. Probe item-list PDFs separately from result PDFs because item quantity can appear there even when the result PDF omits it.
  7. Wrap HTML and PDF fetches in retry/backoff logic so intermittent remote disconnects do not terminate the full run immediately.

Source 2: Etimad

Discovery

Use the public visitor listing URL with filters already embedded in the query.

The listing page is a Vue-powered paginator. The scraper should drive the real browser pager controls, not rely on incrementing the URL alone.

Stop pagination when the visible pager no longer advances, the Next button is disabled, or the listing shows zero tender rows.

Collect STenderId, listing publication date, title, buyer preview, and identifiers.

Award Data Location

The listing card is not enough. Award data is loaded through public detail-page components.

Primary target: GetAwardingResultsForVisitorViewComponenet.

Etimad Endpoints

  • Detail page: https://tenders.etimad.sa/Tender/DetailsForVisitor?STenderId={STenderId}
  • Award results: https://tenders.etimad.sa/Tender/GetAwardingResultsForVisitorViewComponenet?tenderIdStr={STenderId}
  • Dates: https://tenders.etimad.sa/Tender/GetTenderDatesViewComponenet?tenderIdStr={STenderId}
  • Relations/classification: https://tenders.etimad.sa/Tender/GetRelationsDetailsViewComponenet?tenderIdStr={STenderId}

Etimad Parsing Strategy

  1. Load the visitor listing page in a browser session and collect candidate tenders from the visible card set.
  2. Use the rendered paginator's active page and Next button state as the source of truth for traversal.
  3. Open each DetailsForVisitor page.
  4. Call the dates and relations endpoints for notice metadata.
  5. Call the award-results endpoint even if the listing is ambiguous.
  6. Emit one row per awarded supplier row.
  7. Write JSONL incrementally after each tender and checkpoint listing/detail progress so interrupted runs can resume.
Live checks on multiple Etimad public visitor tenders found no item-level tables, BOQ tables, quantity rows, or downloadable item sheets in either the visitor detail tabs or the printable visitor report. Assume Etimad item fields are unavailable from the public visitor flow unless a specific tender exposes an extra public attachment or endpoint.

Shared Workflow

  1. Start from the source discovery URL.
  2. Build a tender queue with source-specific IDs and detail URLs.
  3. Extract notice-level metadata.
  4. Extract award-level data from PDFs or component endpoints.
  5. Check auxiliary item documents or item-level sections when available.
  6. Normalize fields into the shared schema.
  7. Set scraped_at_utc.
  8. Compute dedup_key.

Error Handling

  • If a detail page contains no award/result content, skip row creation.
  • If a NUPCO result PDF exists but parsing fails, retain the tender and PDF URL in an error queue.
  • If an Etimad award-results endpoint returns empty HTML, retain detail metadata for diagnostics.
  • When NUPCO has both result PDFs and item PDFs, merge them by item number where possible to enrich quantity fields.
  • When Etimad award date is missing from the award block, use the competition-related expected award date or award date before leaving it null.
  • For NUPCO, log retry attempts and continue through transient request or PDF-fetch failures where possible.
  • For Etimad, persist a checkpoint and append rows per tender so an interrupted run can resume without losing already-scraped data.
